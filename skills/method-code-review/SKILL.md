# Code Review 方法論

## 核心原則

**從使用者視角追蹤實際路徑，而非開發者視角假設檔案有用。**

不要只看「檔案存在」，要追蹤「檔案是否被使用」。

---

## 審查維度

Code Review 不只是找 bug，更要審視整體品質：

| 維度 | 問自己 |
|------|--------|
| **程式架構** | 模組劃分是否合理？職責是否清晰？ |
| **檔案架構** | 目錄結構是否直觀？找檔案容易嗎？ |
| **可讀性** | 程式碼容易理解嗎？命名清楚嗎？ |
| **維護性** | 未來修改容易嗎？改一處會影響多少地方？ |
| **擴充性** | 新增功能方便嗎？需要大幅重構嗎？ |

**這些維度要持續在 review 過程中自問，而非只在最後檢查一次。**

---

## 檢查清單

### 1. 追蹤實際使用路徑

- HTML 實際載入哪些 `<script src="...">`？
- CSS 實際被哪些頁面引用？
- 有沒有「存在但沒被載入」的死碼？

**方法**：從入口點（HTML）開始追蹤，而非從檔案列表開始假設。

### 2. 跨檔案比對重複邏輯

- 同樣功能是否在多處實作？
- 相同的 utility function 有幾個版本？
- 是否該統一到共用模組？

**方法**：搜尋常見 pattern（`function checkLogin`、`formatDate`），比對實作是否一致。

### 3. 質疑「看起來沒問題」

- 第一輪 review 是否真的夠深入？
- 有沒有只看表面而沒追蹤實際路徑？
- 「應該沒問題」不等於「確認沒問題」

**方法**：自問「我怎麼確定這個檔案有被用？」如果答不出來，就去追蹤。

### 4. 刪除優先於保留

- 死碼就刪，不留「以後可能用到」的僥倖
- 版控會保留歷史，不需要在 codebase 裡留墳墓
- 註解掉的程式碼 = 死碼，一樣刪

### 5. 文件同步更新

- 改完程式碼要更新對應文件
- 確保 skill / README / 註解與現況一致
- 刪除檔案後要檢查有沒有殘留引用
